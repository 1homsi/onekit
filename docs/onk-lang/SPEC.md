# The OneKit Language (`.onk`) — v0.2 Spec

Status: draft. `.onk` is the only schema format the `onek` toolchain accepts.
Nothing here is generated by, checked against, or translated through any
third-party schema compiler - the lexer, parser, IR, and all six code
generators (Go server, Go client, TS client, TS server, Python client, OpenAPI)
are custom, from scratch.

## Goals

- Describe messages, enums, and HTTP services in one file format, with HTTP
  concerns (path, verb, headers, streaming, JSON encoding, validation) as
  first-class decorators.
- No legacy wire-format baggage: no explicit field numbers, no separate
  options-extension mechanism. Fields are just `name: Type`, decorated inline.
- No package registry. Toolchain binaries ship via GitHub Releases. Cross-file
  imports resolve to local paths or plain `github.com/org/repo` paths.

## 1. File layout

```
onekit.toml
models.onk
services/
  user_service.onk
```

`onekit.toml`:

```toml
module = "github.com/1homsi/onekit/examples/simple-api/api"

[generate.go-server]
out = "./api"

[generate.go-client]
out = "./api"

[generate.ts-client]
out = "./web/client"

[generate.openapi]
out = "./docs"
```

## 2. Lexical basics

- Line comments `// ...`, block comments `/* ... */`.
- Identifiers: `[A-Za-z_][A-Za-z0-9_]*`.
- Decorators start with `@`: `@name` or `@name(args)`.
- No trailing punctuation on statements (newline-terminated).

## 3. Packages and imports

```
package examples.simpleapi.models

import "./errors.onk"
import "github.com/1homsi/shared-schemas/billing"
```

`onekit.toml` pins external repo imports:

```toml
[deps]
"github.com/1homsi/shared-schemas" = "v1.4.0"
```

## 4. Scalar types

`string bool int32 int64 uint32 uint64 float32 float64 bytes timestamp`

`Type?` marks a field optional/nullable. `Type[]` marks a field repeated.
`map[K, V]` is a map type.

## 5. Messages

```
message Address {
  street: string
  city: string
}

message User {
  id: string
  name: string @len(2, 100)
  email: string @email
  bio: string? @nullable
  tags: string[]
  labels: map[string, string]
  home_address: Address @flatten(prefix: "home_")
  balance_cents: int64 @encode(number)
}
```

- Field order is `name: Type`, decorators trail on the same line and can
  stack: `field: Type @deco1 @deco2(arg)`.
- No field numbers. Evolution safety (the reason proto forces numbering)
  comes from the compiler diffing the IR between builds instead, not from
  authors manually tracking integers - a `onek check --compat` pass (planned)
  compares two versions of a package and flags removed/retyped fields.

### Oneofs (tagged unions)

```
message Event {
  id: string
  payload: oneof(discriminator: "type", flatten: true) {
    text: TextPayload
    image: ImagePayload
  }
}
```

Each variant's key (`text`, `image`) is both the field name and, by default,
the discriminator tag; override with `@tag("...")` on the variant line.

### Enums

```
enum Status {
  ACTIVE @json("active")
  INACTIVE @json("inactive")
}
```

Values are ordered top-to-bottom (position is the wire/JSON ordinal when
`@encode(number)` is used on a field of this type; default encoding is the
value's name or its `@json` override).

### Map-value / root unwrap

```
message BarList {
  bars: Bar[] @unwrap
}

message UsersResponse {
  users: map[string, User] @unwrap
}
```

### Empty-message, bytes, and timestamp encoding

```
message Response {
  meta: Metadata @empty(preserve)
  hash: bytes @encode(hex)
  created_at: timestamp @encode(unix_seconds)
}
```

`@empty`: `preserve | null | omit`. `@encode` on `bytes`:
`base64 | base64_raw | base64url | base64url_raw | hex`. `@encode` on
`timestamp`: `rfc3339 | unix_seconds | unix_millis | date`.

## 6. Validation

```
message SignupRequest {
  email: string @email
  id: string @uuid
  status: string @in("active", "inactive", "pending")
  age: int32 @range(18, 120)
  name: string @len(2, 100)
}
```

v0.2 ships a fixed decorator set covering string length/pattern/email/uuid/in,
numeric bounds, and required. No expression-language escape hatch yet.

## 7. Services

```
package examples.simpleapi.services

import "./models.onk"

service UserService {
  base_path: "/api/v1"
  headers: {
    "X-API-Key": string @required @format("uuid") @auth("api_key")
  }

  createUser(CreateUserRequest) -> User @post("/users")

  getUser(GetUserRequest) -> User @get("/users/{id}") {
    headers: {
      "X-Request-Id": string @required @format("uuid")
    }
  }

  updateProfile(UpdateProfileRequest) -> Profile @put("/users/{user_id}/profile") @body("profile")

  streamEvents(StreamEventsRequest) -> Event @get("/events") @stream

  searchUsers(SearchUsersRequest) -> SearchUsersResponse @query("/users/search")
}
```

- One line per method: `name(Request) -> Response @verb("/path") [@body("field")] [@stream]`.
  `@verb` is one of `@get @post @put @delete @patch @query` (`@query` is the
  IETF `QUERY` method: safe and idempotent like GET, but carries a body).
- A trailing `{ }` block is only needed for method-level `headers:`.
- `base_path`/`headers:` at the service level apply to every method.
- Query parameters are declared on the request message's fields:

```
message GetPortfolioRequest {
  timeframe: Timeframe @query("timeframe") @required
  account_id: string @query("account_id")
}
```

### Error unions

A method can declare exactly which error types it can return, instead of
relying purely on the `*Error` naming convention at the handler level:

```
getUser(GetUserRequest) -> User | NotFoundError | ValidationError @get("/users/{id}")
```

The type right after `->` is the success type; each `| Type` after it is a
possible error return. Client generators use this to produce exhaustive,
statically-typed error handling (a switch/match over exactly these types)
instead of "parse the body as any `*Error`-shaped JSON." A method with no
`| ...` still allows any `*Error`-suffixed message by convention, matching
today's behavior.

## 8. Errors

Any message whose name ends in `Error` is serialized directly from a
handler's returned error and gets a matching typed exception/interface in
every target language - purely name-based, no decorator needed.

```
message NotFoundError {
  resource_type: string
  resource_id: string
}
```

An error message can pin its HTTP status directly instead of the handler
having to set it manually:

```
message NotFoundError @status(404) {
  resource_type: string
  resource_id: string
}
```

## 9. Doc comments

A `///` line comment (or a run of consecutive `///` lines) immediately above
a `message`, `field`, `enum`, `enum value`, `service`, or `rpc` is that
declaration's doc string. It flows into generated Go doc comments, TS/Python
docstrings, and OpenAPI `description` fields. A plain `//` or `/* */` comment
in between breaks the doc block instead of extending it - it's just a normal,
non-propagated comment.

```
/// A user of the system.
message User {
  /// Unique identifier, assigned server-side.
  id: string
}
```

## 10. Other decorator conventions

These don't need grammar support - they're ordinary decorators the compiler
gives special meaning to:

- `@default(value)` on a field - a non-zero default value, something proto
  never allowed.
- `@deprecated("use X instead")` on anything - surfaces as `// Deprecated:`
  in Go, `@deprecated` in TS, and `deprecated: true` in OpenAPI.
- `@status(code)` on an `*Error` message - the HTTP status the error
  serializes with, without handler code having to set it.
